### Логика тестовых данных

* Решил немного заморочиться и для пущей реалистичности сделал генерацию комбинаций из реальных русских имён и фамилий: 55 мужских имён, 60 женских имён, 30 фамилий. Итого (55 + 60) * 30 = 3450 уникальных комбинаций. Пользователей сделал 80 тысяч. Специально для столбца `gender` сделал enum тип `gender_type` с двумя значениями: `male` и `female`.

* В качестве регионов взял 10 самых крупных городов России. Для таблицы специализаций скопировал все 26 IT специализаций из фильтра поиска на hh. Работодателей сделал 500 штук с названиями типа "Рандомная компания N".

* В вакансиях и резюме все столбцы заполнял рандомными значениями из возможных. Вакансий сделал 10 тысяч, резюме — 100 тысяч. В столбец `title` и там, и там запихивал полученное по id название специализации, чтобы было поинтереснее. Значения `salary_from` выбирал по принципу "округлить(рандом([0, 1)) * 300000, <адекватное кол-во знаков>)". Побочный прикол: встречаются записи, где это значение равно нулю. Значения `salary_to` выбирал по принципу "salary_from + округлить(рандом([0, 1)) * 30000, -3)". Для резюме столбец `salary_to` не делал. Не представляю человека, который бы хотел себе потолок зарплаты :)  В вакансиях для столбцов `required_experience` и `payment_frequency` создал enum типы `required_experience_type` и `payment_frequency_type`. Второй по сути фиктивный и появился потому, что я ориентировался на hh. Первый же используется для мэтча вакансий и резюме и имеет значения `no`, `from_1_to_3`, `from_3_to_6`, `more_than_6` — всё как на hh.

* Даты публикации вакансий и резюме брал в диапазоне от даты генерации данных до этого дня 3 года назад. Сделал временной интервал побольше, чтобы было поинтереснее. Как и просилось в конце лекции и в чате в ММ.

* Отклики создавал по следующему принципу. Для каждого резюме брал список всех подходящих по региону, специализации, опыту и минимальной зарплате вакансий, которые были опубликованы раньше, чем резюме, и делал на них отклик.

* Даты откликов брал по принципу "дата публикации резюме + рандом([0, 1)) * минимум(1 месяц, дата генерации данных - дата публикации резюме)". Сделал так, чтобы отклик не мог быть сделан раньше, чем опубликовано резюме. Максимальный интервал в один месяц, чтобы сильно не размазывать отклики. Выбор минимума — дла того, чтобы не было откликов из будущего.

* Все текстовые столбцы, кроме `title` и `<объект>_name`, заполнял одинаковыми заглушками.
